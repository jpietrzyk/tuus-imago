# Plan: Decap CMS for Legal Pages

## Goal

Add Decap CMS to manage content for ~12 legal/static pages and their navigation menu order. Content stored as markdown files in Git, requiring a Netlify rebuild to publish changes.

---

## Architecture Decisions (confirmed)

- **Content format**: One `.md` file per page in `content/legal/` directory with frontmatter metadata + markdown body
- **Page rendering**: Single shared `LegalPageLayout` component that renders any legal page from its markdown
- **Content scope**: ALL legal page text managed by CMS; i18n JSON files retain only non-legal strings (common, checkout, upload, etc.)
- **Navigation order**: Frontmatter `menuOrder` field controls order in the legal navigation drawer

---

## Current State

- 12 legal pages as hardcoded JSX in `src/pages/` (~2,617 lines total)
- All text lives in `src/locales/pl.json` and `src/locales/en.json` (1002 lines each)
- Navigation links hardcoded in `src/components/legal-navigation-sheet.tsx`
- Each page reimplements the same shell (back link, title, card wrapper, sections)
- No markdown processing or CMS integration exists
- Language is hardcoded to Polish

## Implementation Steps

### Step 1: Install dependencies

```bash
pnpm add vite-plugin-decap-cms -D
pnpm add react-markdown remark-gfm
```

- `vite-plugin-decap-cms` ‚Äî auto-generates admin panel at `/admin/`
- `react-markdown` ‚Äî renders markdown to React components
- `remark-gfm` ‚Äî GitHub-flavored markdown support (tables, lists, etc.)

Frontmatter fields:
- `title` ‚Äî page heading
- `subtitle` ‚Äî page subtitle
- `slug` ‚Äî URL slug (matches route)
- `icon` ‚Äî lucide-react icon name (string, mapped at render time)

- `menuOrder` ‚Äî sort order within section (number)
- `lastUpdated` ‚Äî date string shown in page footer
- `lang` ‚Äî language code (`pl` currently; future-proof for EN)

### Step 4: Create Decap CMS config

**File:** `public/admin/config.yml`

Configure Decap CMS with:
- **File collection** for each legal page markdown file
- Fields: title, subtitle, slug, icon, menuSection, menuOrder, lastUpdated, body (markdown)
- Git Gateway backend (Netlify OAuth)
- Editorial workflow disabled (simple publish flow)

```yaml
backend:
  name: git-gateway
  branch: main

media_folder: "content/media"
public_folder: "/content/media"

collections:
  - name: "legal"
    label: "Legal Pages"
    folder: "content/legal"
    create: false
    extension: "md"
    format: "frontmatter"
    fields:
      - { name: "title", label: "Title", widget: "string" }
      - { name: "subtitle", label: "Subtitle", widget: "string" }
      - { name: "slug", label: "URL Slug", widget: "string" }
      - { name: "icon", label: "Icon", widget: "select", options: [FileText, Shield, Cookie, CheckCircle, Lock, RotateCcw, Truck, AlertCircle, BadgeDollarSign, Building2, Mail] }
      - { name: "menuSection", label: "Menu Section", widget: "select", options: [legal, payments, company] }
      - { name: "menuOrder", label: "Menu Order", widget: "number", value_type: int }
      - { name: "lastUpdated", label: "Last Updated", widget: "datetime", format: "YYYY-MM-DD" }
      - { name: "lang", label: "Language", widget: "hidden", default: "pl" }
      - { name: "body", label: "Content", widget: "markdown" }
```

### Step 5: Build-time content loading

**New file:** `src/lib/content-loader.ts`

Use Vite's `import.meta.glob` to load all markdown files at build time:

```ts
const rawPages = import.meta.glob('/content/legal/*.md', { eager: true, query: '?raw', import: 'default' })
```

Parse frontmatter (use a small frontmatter parser like `gray-matter` or a ~20-line regex parser) and export:
- `getPageBySlug(slug: string)` ‚Äî returns parsed page data
- `getAllPages()` ‚Äî returns all pages sorted by menuOrder
- `getPagesBySection(section: string)` ‚Äî returns pages filtered by menuSection

This approach:
- Content is bundled at build time (no runtime fetch)
- No extra dependencies needed beyond a frontmatter parser
- Works with Vite's static analysis

### Step 6: Create shared LegalPageLayout component

**New file:** `src/components/legal-page-layout.tsx`

Generic layout that renders any legal page:

```tsx
interface LegalPageLayoutProps {
  title: string
  subtitle: string
  lastUpdated: string
  markdownContent: string
}
```

- Renders the shared page shell (back link, centered title, card wrapper)
- Uses `react-markdown` to render the markdown body
- Applies Tailwind prose-like styling for headings, lists, tables, links
- Shows "Last updated: {date}" footer

### Step 7: Create markdown styling

**New file:** `src/components/markdown-styles.css`

Tailwind prose-style classes for rendered markdown:
- Headings (h1-h4) with proper sizing and spacing
- Paragraphs, lists (ul/ol), tables
- Links, bold, italic
- Consistent with existing Card/CardContent styling

Alternatively, install `@tailwindcss/typography` and use the `prose` class.

### Step 8: Refactor legal pages to use content loader

For each of the 12 legal pages, replace the hardcoded JSX with a thin wrapper:

**Before (e.g., `src/pages/terms.tsx`):**
```tsx
// 321 lines of hardcoded JSX with t() calls
export function TermsPage() {
  return (
    <div className="min-h-screen bg-gray-50">
      {/* ... hundreds of lines ... */}
    </div>
  )
}
```

**After:**
```tsx
import { LegalPageLayout } from '@/components/legal-page-layout'
import { getPageBySlug } from '@/lib/content-loader'

export function TermsPage() {
  const page = getPageBySlug('terms')
  if (!page) return null
  return (
    <LegalPageLayout
      title={page.title}
      subtitle={page.subtitle}
      lastUpdated={page.lastUpdated}
      markdownContent={page.body}
    />
  )
}
```

Each page component becomes ~10 lines.

### Step 9: Make navigation menu data-driven

**File:** `src/components/legal-navigation-sheet.tsx`

Replace the hardcoded `legalLinks`, `paymentLinks`, `companyLinks` arrays with data from the content loader:

const legalLinks = getPagesBySection('legal')

Each page's frontmatter provides `slug`, `title`, `icon`, and `menuOrder`. The icon string is mapped to the actual lucide-react component via a lookup object.

## Special Cases

- **Complaint page**: Form UI stays in JSX, only informational content goes to CMS
- **Contact page**: May have interactive elements (email links, phone numbers) ‚Äî keep in markdown with HTML anchors
- **External P24 links** in navigation: These stay hardcoded in the nav component (not CMS content)

## What CMS Editors Can Do After Implementation

1. Navigate to `https://yoursite.netlify.app/admin/`
2. Log in via GitHub (Netlify Git Gateway)
3. Edit any legal page's text, title, subtitle, or icon
4. Change menu ordering via the `menuOrder` number field
5. Save ‚Üí creates a Git commit ‚Üí triggers Netlify rebuild ‚Üí publishes changes

## Limitations

- Content changes require a full Netlify rebuild (~2 minutes)
- Only one language supported initially (Polish); `lang` frontmatter field is future-proofing
- The `/admin/` panel is a separate React app (not embedded in your main app)
- No real-time preview of your actual page design in the CMS editor

# HubSpot CRM Integration Plan

## Goal

When a user submits the checkout form, push their data to **HubSpot CRM** as a Contact so the sales team can contact them — regardless of whether the Przelewy24 payment succeeds.

.

### Why HubSpot over Zoho?

 Zoho CRM Free Edition does **not** support custom fields, which means order-specific data (order number, value, items) would be crammed into a `Description` text block. HubSpot Free CRM supports:
 custom properties on Contacts, unlimited contacts, unlimited users, REST API, and a simpler authentication model (static tokens from the new Developer Platform).

---

## Architecture

**Dedicated Netlify function + frontend call pattern.**
```
[checkout form submit]
       │
       ▼
create-order (existing) ──── Supabase: saves order
       │
       ▼  (after order creation succeeds)
sync-hubspot-contact (NEW function) ──── HubSpot CRM: upserts Contact
       │
       ▼  (after HubSpot sync, fire-and-forget style)
create-przelewy24-session (existing) ──── P24: payment session
```

The frontend calls `sync-hubspot-contact` between order creation and P24 session creation. HubSpot failure does **not** block the checkout flow — it's logged but not fatal.
### Why this pattern?
- **Decoupled** — separate function, separate concerns, testable in isolation
- **Non-blocking** — HubSpot errors don't prevent order creation or payment
- **Frontend orchestration** — simple to implement, matches existing `createOrder` → `createP24Session` chaining pattern
- **Idempotent** — uses email-based upsert, safe to retry

## Data mapping: Checkout Form → HubSpot Contact

HubSpot uses **Contacts** (not Leads — Leads require Sales Hub Professional/Enterprise). The free plan supports unlimited custom properties via API.
### Standard Properties (direct mapping)
| Checkout Field | DB Column (`orders`) | HubSpot Property | Type | Notes |
|---|---|---|---|---|
| `name` | `customer_name` | `firstname` + `lastname` | string | Split on last space. `lastname` is effectively required for display. |
| `email` | `customer_email` | `email` | string | Primary unique identifier |
| `phone` | `customer_phone` | `phone` | string | Optional |
| `address` | `shipping_address` | `address` | string | |
| `city` | `shipping_city` | `city` | string | |
| `postalCode` | `shipping_postal_code` | `zip` | string | |
| `country` | `shipping_country` | `country` | string | ISO 2-letter code |
### Custom Properties (created via API or HubSpot UI)
These custom properties must be created in HubSpot before or during first deployment. They can be created via the Properties API (`POST /crm/v3/properties/contacts`) or manually in **Settings -> Properties**:
| Internal Name | Label | Type | fieldType | Description |
|---|---|---|---|---|
| `order_number` | Order Number | `string` | `text` | Tuus Imago order number (e.g. TI-2026-000001) |
| `order_value` | Order Value | `number` | `number` | Total order value in PLN |
| `items_count` | Items Count | `number` | `number` | Number of canvas prints ordered |
| `checkout_date` | Checkout Date | `datetime` | `date` | ISO 8601 datetime of checkout |
| `marketing_consent` | Marketing Consent | `bool` | `booleancheckbox` | Whether user gave marketing consent |
| `order_status` | Order Status | `string` | `text` | Current order status (pending_payment, paid, etc.) |

**Property group:** Create a property group called `tuus_imago` to organize these custom properties.
### Auto-set Properties
| HubSpot Property | Value |
|---|---|
| `lead_source` (if available) or custom `lead_source` | "Website Checkout" |
| `lifecycle_stage` | `lead` |

## Authentication Strategy

HubSpot Developer Platform uses **project-based apps with static access tokens**. Legacy private apps are deprecated — new integrations should use the project-based approach.

### Setup: Create a Project App (recommended approach)
1. Install HubSpot CLI: `npm install -g @hubspot/cli@latest`
2. Authenticate: `hs account auth`
3. Create project: `hs project create --platform-version 2026.03`
   - Select **App** template
   - Choose **Private** distribution
   - Choose **Static token** authentication
   - Add scopes: `crm.objects.contacts.read`, `crm.objects.contacts.write`, `crm.schemas.contacts.read`
4. Upload project: `hs project upload`
5. Copy the **Access Token** from **HubSpot → Development → Projects → your app → Auth tab**
6. Store it as Netlify env var `HS_PRIVATE_APP_ACCESS_TOKEN`

**Alternative (simpler, beta): Service Key**
If you don't want to use the CLI, HubSpot also offers **Service Keys** (beta):
1. Go to **HubSpot → Development → Keys → Service Keys**
2. Click **Create service key**
3. Add scopes: `crm.objects.contacts.read`, `crm.objects.contacts.write`
4. Copy the key
5. Store as Netlify env var `HS_PRIVATE_APP_ACCESS_TOKEN`

Both options produce a static `Bearer` token — same usage pattern, no refresh needed.
### Environment variables needed:
| Variable | Description |
|---|---|
| `HS_PRIVATE_APP_ACCESS_TOKEN` | Static access token from HubSpot project app or service key |

That's it. **One token, no refresh logic, no OAuth flow.** The token doesn't expire unless manually revoked.
### API Endpoint Pattern
```
PATCH https://api.hubapi.com/crm/v3/objects/contacts/{email}?idProperty=email
Headers:
  Authorization: Bearer {HS_PRIVATE_APP_ACCESS_TOKEN}
  Content-Type: application/json
Body:
  { "properties": { "email": "...", "firstname": "...", ... } }
```
### Code changes needed (env var rename)
The existing code uses `HUBSPOT_ACCESS_TOKEN`. The following files need the env var renamed to `HS_PRIVATE_APP_ACCESS_TOKEN`:
- `netlify/functions/_shared/hubspot.ts` — `getHubSpotConfig()` reads the env var
- `netlify/functions/sync-hubspot-contact.ts` — no direct change (uses `getHubSpotConfig()`)
- `netlify/__tests__/_shared/hubspot.test.ts` — test env var setup
- `netlify/__tests__/sync-hubspot-contact.test.ts` — test env var setup
- `.env.example` — env var name
- `README.md` — documentation references

No other code changes needed — the `Bearer` token pattern in API calls stays the same.

## Implementation Steps

### Step 1: HubSpot CRM Account & Setup

Manual steps:
- Create a free HubSpot CRM account at https://www.hubspot.com/products/crm
- **Option A (recommended):** Create a project app via CLI (see Authentication Strategy above)
- **Option B (simpler, beta):** Create a service key in HubSpot → Development → Keys → Service Keys
- Copy the access token
- Set `HS_PRIVATE_APP_ACCESS_TOKEN` env var in Netlify
- Custom properties are auto-created on first sync (see Step 2), or create manually in **Settings → Properties**

### Step 2: Shared HubSpot Utility Module
**File:** `netlify/functions/_shared/hubspot.ts`

Following the pattern of `_shared/przelewy24.ts`, create:
```
Exports:
- HubSpotConfig (type) — config shape from env vars
- getHubSpotConfig() — reads HS_PRIVATE_APP_ACCESS_TOKEN env var, validates
- HubSpotContactPayload (type) — Contact properties shape for API
- upsertHubSpotContact(properties) — PATCH /crm/v3/objects/contacts/{email}?idProperty=email
- ensureCustomProperties() — creates custom properties if they don't exist (idempotent)
- buildContactProperties(order) — maps order row to HubSpot Contact properties
```

**Key implementation details:**
- Use native `fetch()` (same as P24 integration)
- Authorization header: `Bearer {HS_PRIVATE_APP_ACCESS_TOKEN}`
- Upsert via `PATCH /crm/v3/objects/contacts/{email}?idProperty=email` — creates if not exists, updates if exists
- Custom property creation via `POST /crm/v3/properties/contacts` — skip if already exists (409)
- Property group creation via `POST /crm/v3/properties/contacts/groups` — named `tuus_imago`
- No token refresh needed — static token

### Step 3: Netlify Function — `sync-hubspot-contact`

**File:** `netlify/functions/sync-hubspot-contact.ts`
```
Handler flow:
1. Validate POST method
2. Parse body: { orderId: string }
3. Initialize Supabase client (service role)
4. Fetch order by ID from Supabase (with all fields)
5. If order not found → 404
6. Build HubSpot Contact properties from order data
7. Call upsertHubSpotContact()
8. On success → update order row with hubspot_contact_id + hubspot_synced_at, return 200
9. On failure → log error, return 200 anyway (non-blocking)
```

**Important:** The function returns 200 even on HubSpot failure. This ensures the checkout flow is never blocked. HubSpot errors are logged via `console.error` for monitoring.

## Risk Mitigation

| Risk | Mitigation |
|---|---|
| HubSpot API downtime | Non-blocking — function returns 200 even on failure |
| Duplicate contacts | Use PATCH with `idProperty=email` for upsert behavior |
| Rate limiting | Single Contact per checkout — well within HubSpot limits (100 req/sec) |
| Token revocation | Admin can regenerate token and update env var |
| Name parsing edge cases | Fallback: if no space in name, put full name in `lastname`, leave `firstname` empty |
| Custom property creation failure | Non-fatal — contact is still created with standard properties; custom properties can be added later |

# Supabase Auth & Customer Account Integration Plan

## Overview

Integrate Supabase Auth into the existing Tuus Imago canvas-print e-commerce SPA so customers can optionally create accounts to view order history, manage saved addresses, apply promo codes, and see payment history. Guest checkout remains fully functional.

**Auth methods**: Email + password, Google OAuth, Facebook OAuth, Magic link (email)
**Coupon system**: Admin-created promo codes applied at checkout
**Payment data**: Payment history display only (no saved payment methods)

---

## Phase 1: Database Schema (Supabase Migrations)

### 1.1 Auth foundation — link orders to `auth.users`

**Migration: `202604070001_add_user_id_to_orders.sql`**

```sql
-- Add nullable user_id to orders (nullable because guest checkout persists)
alter table public.orders
  add column if not exists user_id uuid references auth.users(id) on delete set null;

create index if not exists orders_user_id_idx on public.orders(user_id)
  where user_id is not null;
```

Orders placed by guests have `user_id = null`. When a guest later creates an account with the same email, a backfill function links their previous orders.

### 1.2 Customer profiles table

**Migration: `202604070002_create_profiles.sql`**

```sql
create table public.profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  full_name text,
  phone text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

alter table public.profiles enable row level security;

create policy "Users can read own profile"
  on public.profiles for select
  using (auth.uid() = id);

create policy "Users can update own profile"
  on public.profiles for update
  using (auth.uid() = id);

-- Auto-create profile on signup
create function public.handle_new_user()
returns trigger
language plpgsql
security definer set search_path = public
as $$
begin
  insert into public.profiles (id, full_name)
  values (new.id, coalesce(new.raw_user_meta_data->>'full_name', ''));
  return new;
end;
$$;

create trigger on_auth_user_created
  after insert on auth.users
  for each row execute function public.handle_new_user();
```

### 1.3 Saved addresses table

**Migration: `202604070003_create_addresses.sql`**

```sql
create table public.addresses (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  label text not null default 'Home',          -- e.g. "Home", "Office"
  name text not null,
  phone text,
  address text not null,
  city text not null,
  postal_code text not null,
  country text not null,
  is_default boolean not null default false,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

alter table public.addresses enable row level security;

create policy "Users can CRUD own addresses"
  on public.addresses for all
  using (auth.uid() = user_id)
  with check (auth.uid() = user_id);

create index if not exists addresses_user_id_idx on public.addresses(user_id);
```

### 1.4 Promo codes table

**Migration: `202604070004_create_coupons.sql`**

```sql
create table public.coupons (
  id uuid primary key default gen_random_uuid(),
  code text not null unique,
  description text,
  discount_type text not null check (discount_type in ('percentage', 'fixed_amount')),
  discount_value numeric(10,2) not null check (discount_value > 0),
  currency text not null default 'PLN',
  min_order_amount numeric(10,2),
  max_uses integer,
  used_count integer not null default 0,
  valid_from timestamptz,
  valid_until timestamptz,
  is_active boolean not null default true,
  created_at timestamptz not null default now()
);

alter table public.coupons enable row level security;

-- Only service role can manage coupons; public can only read active ones
create policy "Public can read active coupons"
  on public.coupons for select
  using (is_active = true);

-- Track per-user coupon usage
create table public.coupon_usages (
  id uuid primary key default gen_random_uuid(),
  coupon_id uuid not null references public.coupons(id),
  user_id uuid references auth.users(id) on delete set null,
  order_id uuid not null references public.orders(id),
  used_at timestamptz not null default now()
);

alter table public.coupon_usages enable row level security;

create index if not exists coupon_usages_coupon_id_idx on public.coupon_usages(coupon_id);
create index if not exists coupon_usages_user_id_idx on public.coupon_usages(user_id);
```

### 1.5 Add coupon fields to orders

**Migration: `202604070005_add_coupon_to_orders.sql`**

```sql
alter table public.orders
  add column if not exists coupon_id uuid references public.coupons(id),
  add column if not exists coupon_code text,
  add column if not exists discount_amount numeric(10,2) not null default 0;
```

## Phase 3: Netlify Functions (Backend)

### 3.1 Link guest orders on signup

**New file: `netlify/functions/on-user-signup.ts`** (or use Supabase webhook / database trigger)

Alternative (simpler): a Supabase database trigger that runs after `auth.users` insert and updates all matching `orders.user_id` where `orders.customer_email = new.email AND orders.user_id IS NULL`.

```sql
-- Add to migration 1.2 or a new migration
create function public.link_guest_orders()
returns trigger
language plpgsql
security definer set search_path = public
as $$
begin
  update public.orders
  set user_id = new.id
  where customer_email = new.email
    and user_id is null;
  return new;
end;
$$;

create trigger on_auth_user_link_orders
  after insert on auth.users
  for each row execute function public.link_guest_orders();
```

### 3.2 Validate & apply coupon

**New file: `netlify/functions/validate-coupon.ts`**

- `POST { code, orderTotal }` → validates coupon is active, within date range, under max uses, meets minimum order amount
- Returns `{ valid, couponId, code, discountType, discountValue, discountAmount }`
- Uses service role key (bypasses RLS for admin-configured coupons)

### 3.3 Update `create-order.ts`

- Accept optional `couponCode` in the payload
- If provided, validate coupon, compute discount, store `coupon_id`, `coupon_code`, `discount_amount` on the order
- Update `total_price` to reflect the discount
- Increment `coupons.used_count` and insert into `coupon_usages`
- If user is authenticated (pass `userId` from frontend), set `user_id` on the order

### 3.4 Customer orders endpoint

**New file: `netlify/functions/customer-orders.ts`**

- `GET` — returns all orders for the authenticated user
- Validates the user's JWT from the `Authorization: Bearer` header
- Returns orders with items, payment status, shipment status, coupon info

### 3.5 Customer addresses CRUD

**New file: `netlify/functions/customer-addresses.ts`**

- `GET` — list user's addresses
- `POST` — add address
- `PATCH` — update address
- `DELETE` — remove address
- All operations validate the JWT and filter by user ID

Alternatively, since RLS policies are already in place, the frontend could use the Supabase client directly for addresses. The serverless function approach is more consistent with the existing architecture where all DB access goes through Netlify functions.

## Phase 9: Supabase Dashboard Configuration (Manual)

These steps are performed in the Supabase dashboard, not in code:

1. **Enable Auth providers**:
   - Email (enabled by default)
   - Google OAuth — configure OAuth client ID + secret from Google Cloud Console
   - Facebook OAuth — configure App ID + secret from Meta for Developers
   - Magic Link — enable "Enable Email Signup" and confirm email settings

2. **Configure Auth URL redirects**:
   - Site URL: `https://tuusimago.com` (or Netlify deploy URL)
   - Redirect URLs: `https://tuusimago.com/auth/callback`

3. **Email templates** (optional):
   - Customize confirmation email, password reset email, magic link email

4. **Environment variables** (Netlify):
   - Add `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` to Netlify env

## Key Architectural Decisions

1. **Supabase client on frontend**: The `VITE_SUPABASE_ANON_KEY` is safe to expose — it only works with RLS policies. All privileged operations still go through Netlify functions using `SUPABASE_SECRET_KEY`.

2. **Serverless-first for data mutations**: Following the existing pattern, all writes (create order, apply coupon, manage addresses) go through Netlify functions. The frontend Supabase client is primarily used for auth state management and optional direct reads of user-scoped data.

3. **Nullable `user_id` on orders**: Keeps guest checkout working. The DB trigger `link_guest_orders()` retroactively links past orders when a guest creates an account with the same email.

4. **Coupon validation is server-side**: Promo codes are validated in the backend to prevent tampering. The frontend only displays the result.

5. **No stored payment methods**: Payment history is read-only, derived from existing order data. No sensitive payment tokens are stored.

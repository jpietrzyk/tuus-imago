### Phase 2: Fix Known Issues

**2.1 Payment return screen shows raw orderId instead of orderNumber**
- File: `src/pages/checkout.tsx:621-627`
- Currently shows `returnOrderId` (UUID) instead of the human-readable `TI-YYYY-NNNNNN` order number
- Fix: Look up order by orderId from the URL param, or pass orderNumber via return URL

**2.2 Coupon discount not passed to P24 session**
- File: `netlify/functions/create-przelewy24-session.ts:156-158`
- Uses `total_price` from DB (which should include discount if `create-order` sets it correctly)
- Verify `create-order` sets `total_price = base_price - discount_amount` when coupon is applied
- If total_price in DB is pre-discount, fix to use the discounted amount

**2.3 Shipping cost handling**
- File: `netlify/functions/create-przelewy24-session.ts:157,179`
- `shipping` is sent to P24 register as separate field
- Verify P24 amount = items total (not items + shipping) and shipping is sent separately
- Currently: `amount = toMinorUnits(order.total_price)` and `shipping = toMinorUnits(order.shipping_cost)`
- This means total_price should NOT include shipping (P24 adds it). Verify DB schema.

### Phase 3: Error Handling & Resilience Improvements

**3.1 Add better P24 error response logging**
- File: `netlify/functions/create-przelewy24-session.ts:213-220`
- Log `responseCode` from P24 response for debugging (e.g. error code 101 = invalid credentials)
- Include P24 response code in error stored in `payment_error` column

**3.2 Store payment_error on failed registration**
- Currently when P24 register fails, we return 502 but don't store `payment_error` on the order
- Fix: update order with `payment_status: "failed"` and `payment_error: <message>`

**3.3 Webhook timeout / retry handling**
- P24 retries webhook on failure (up to 5 retries within ~30 minutes)
- Verify our webhook returns proper status codes (currently correct: 400 for bad sign, 200 for success)
- Consider adding idempotency check for notification processing (already exists for verified orders)

**3.4 Frontend payment status polling**
- File: `src/pages/checkout.tsx` payment return screen
- Currently shows static "payment pending" - user must manually refresh
- **Implement polling**: every 5 seconds, call `customer-orders?orderId=<id>` to check payment status
- When `status === "paid"`, replace pending screen with success confirmation (green checkmark, order number, "Back to home")
- When `payment_status === "failed"`, show retry option
- Add polling with `useEffect` + `setInterval`, clean up on unmount
- Stop polling after 5 minutes (timeout) with a message to check order status later

### Phase 4: Production Readiness Checklist

**4.1 Environment variable migration for production**
- When going live: change `P24_API_BASE_URL` from sandbox to `https://secure.przelewy24.pl/api/v1`
- All other credentials will change to production values from P24 merchant panel
- Ensure `.env.example` documents both sandbox and production URLs

**4.2 Netlify deployment configuration**
- Add P24 env vars to Netlify site settings (or netlify.toml `[context.production]`)
- Set `P24_STATUS_URL` to production webhook URL
- Ensure `SITE_URL` points to production domain

**4.3 IP whitelist for P24 webhook (optional)**
- P24 can send notifications from specific IPs
- Consider adding IP validation in webhook handler for production
- P24 provides IP list in their documentation

**4.4 Payment success email/notification**
- Currently no email notification when payment is verified
- Consider adding email trigger in webhook handler after successful verification
- Could integrate with HubSpot (already connected) for transactional emails
